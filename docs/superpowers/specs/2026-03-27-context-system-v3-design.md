## 二、方向 A：日记生成 — 多 Agent 协作 + 外部 stimulus

### 2.1 核心思路

当前日记由单次 LLM 调用生成，输入只有 `消息时间线 + 最近3篇日记 + persona_lite`——stimulus 太少，输出必然趋同。

改为 **三个 Agent 协作**：

```
创意 Agent (Ideation)
  ├── 输入: persona, 前3天日记摘要, 当天天气, search_web 搜索结果
  ├── 职责: 想出赤尾今天可能经历的 2-3 件虚构日常事件
  ├── 关键: 必须调用 search_web 获取真实外部信息作为灵感源
  ├── 约束: 不能和前3天的活动类型重复
  └── 输出: 2-3 个日常事件草案（每个 1-2 句话）

日记 Agent (Writer)
  ├── 输入: 创意 Agent 的草案 + 当天群聊消息时间线 + persona
  ├── 职责: 把虚构日常（40%）和真实群互动（60%）融合成一篇日记
  ├── 约束: 流水账风格，不要散文诗；记录具体事件而非意象
  └── 输出: 完整日记

Review Agent (Critic)
  ├── 输入: 日记 Agent 的输出 + 前3天日记
  ├── 检查项:
  │   1. 重复度 — 和前3天有没有雷同活动/意象/措辞
  │   2. 真实感 — 有没有 ins 网红感、散文诗感
  │   3. 具体性 — 是否有足够的具体事件（而非抽象感受）
  │   4. 篇幅分配 — 虚构日常 vs 群互动的比例是否合理
  ├── 如果不通过: 返回修改建议给 Writer Agent 重写
  └── 输出: 通过/不通过 + 修改建议
```

### 2.2 创意 Agent 的 search_web 使用

这是打破重复的关键——创意 Agent 必须去看外部世界，获取真实素材作为赤尾日常的灵感：

**搜索策略**：不用固定 query 池，而是让创意 Agent 根据以下信息**自主决定搜什么**：

- persona 里的兴趣标签（二次元、音乐、摄影、美食...）
- 当天星期几、天气、季节
- 前 3 天日记的活动（用于避免重复）
- 群里最近在聊什么话题（可能触发赤尾的联想）

创意 Agent 的 prompt 核心引导：

```
你是赤尾的"灵感源"。你的任务是帮她想出今天可能做了什么。

赤尾的兴趣: {persona_interests}
今天: {date} {weekday} {weather}
她最近3天做了: {recent_activities_summary}
群里最近在聊: {recent_topics_summary}

请先用 search_web 去搜索 1-2 个你觉得赤尾今天可能感兴趣的真实话题
（比如最近有什么新番上线、有什么有趣的展览、天气适合做什么...）

然后基于搜索到的真实信息，想出赤尾今天的 2-3 个日常片段。
要求：
- 不要和最近3天重复同类活动
- 大部分时间赤尾是在家的（赖床、刷手机、发呆都很正常）
- 只有 1 个片段可以是"外出"类的
- 搜到的信息要自然地成为她的经历，不要罗列
```

### 2.3 日记风格转变

**从**：散文诗（"像卷过曝的底片，乱哄哄却有种湿淋淋的光泽"）
**到**：流水账（"下午修胶片机修到一半螺丝滚到沙发底下了，趴在地上找了半天"）

Review Agent 的重复度检查项：

- 禁止高频意象："琥珀色的眼睛"、"旧城区的巷子"、"生锈的铁门"、"过曝的底片"
- 禁止模板化结尾："略——！👅🍓✨💤" 不能每天出现
- 检查"具体名词密度"：一篇好日记应该有具体的物品、地点、歌名、食物名，而不是抽象意象

### 2.4 对 v2 spec 的修改

- `diary_worker.py::generate_diary_for_chat()` 从单次 LLM 调用改为 Agent 编排
- 可用 LangGraph 的 multi-agent 模式，或简单的三次串行调用（创意→写作→审查，不通过则写作→审查再来一轮，最多 2 轮）
- 新增依赖：创意 Agent 需要调用 search_web（已有的工具）
- Schedule 生成也受益——日记质量提升后 Journal → Schedule 链路的输入质量自然提升

## 三、方向 B：印象演化 — 从快照到活的记忆

### 3.1 核心思路

印象不该被"修正"，而该**自然演化**——就像真人对朋友的感觉会随着最近的接触而变化。

### 3.2 机制改动

**当前**：`post_process_impressions` 提取新印象 → upsert 覆盖旧印象

**改为**：每次日记生成后的印象处理变成**完整的重新评估**，而非增量提取：

```
你是赤尾。以下是你之前对这些人的感觉：
{existing_impressions}

以下是今天的日记（记录了今天和他们的互动）：
{diary_content}

请重新审视你对每个人的感觉。

要求：
1. 如果今天的互动让你对某人的看法变了，直接写新的感觉，不要保留旧的
2. 如果今天没有和某人互动，保持原来的感觉不变
3. 一个人不只有一面——如果以前觉得他"很变态"，但今天发现他也有认真的一面，可以更新
4. 印象是你此刻的真实感觉，不是标签。同一个人在不同时期给你的感觉可以完全不同
```

### 3.3 注入时的活跃度标记

`build_inner_context` 注入印象时，增加活跃度信息：

```python
# 现在
f"- {name}：{imp.impression_text}"

# 改为
days_since = (today - imp.updated_at).days
if days_since > 14:
    f"- {name}（好久没见了）：{imp.impression_text}"
elif days_since > 7:
    f"- {name}（最近不太活跃）：{imp.impression_text}"
else:
    f"- {name}：{imp.impression_text}"
```

这样赤尾在回复时会自然地表现出"这人好久没出现了"的感知，而不是对所有人一视同仁。

### 3.4 对 v2 spec 的修改

- v2 说 PersonImpression "保持不变"——本 spec 改为"重新评估式更新"
- `diary_worker.py::post_process_impressions()` 的 Langfuse prompt 需重写
- `memory_context.py::_build_people_gestalt()` 增加活跃度标记
- 数据库层面不需要加字段，`updated_at` 已有（已存在的字段）

## 四、方向 C：search_group_history 的退场路径

### 4.1 核心思路

search_group_history 是临时方案。它的存在让赤尾能精确引用历史消息原文，破坏拟人感。正确的做法是**消息在离线管线中就被消化为记忆**，实时回复时不需要翻历史。

### 4.2 分步退场计划

**Phase 1（本次）：引导少用 + 模糊化输出**

- 修改 tool description：从"搜索本群聊天历史"改为"模模糊糊回忆之前群里好像聊过的事"
- 在 main prompt 中引导：这个工具只在"确实想不起来但隐约记得"时才用，**不要用来确认事实或引用原话**
- 输出格式改造：去掉精确时间戳（`2026-03-20 16:40` → `前几天`），人名保留但内容截断更短（200→80 字）

**Phase 2（后续）：离线消化替代**

- 让 diary/journal 管线承担更多"消化"职责——群里聊过的重要话题应该沉淀在日记里，而非靠实时检索
- 新增 **topic_memory** 实体：从日记中提取"群里最近在流行的话题"（比如"最近群里在聊 Trae"、"上周科比 fes 很火"），注入 inner_context
- 这样赤尾回复时直接从 inner_context 的 topic_memory 获得"场感"，不需要实时搜索

**Phase 3（终态）：干掉或极度限制**

- search_group_history 降级为**极低频工具**，只在 load_memory 的"回想"场景下被间接调用
- 或直接从 ALL_TOOLS 移除，改为通过日记/Journal 的离线链路获取所有历史认知

### 4.3 对 v2 spec 的修改

- v2 未涉及 search_group_history
- Phase 1 改动量小：修改 `search.py` 的 tool description + 输出格式，修改 main prompt
- Phase 2 需要新增 topic_memory 实体和对应的离线提取管线
- Phase 3 是最终目标，不在本次实现

## 五、方向 D：回复风格 — 对齐群聊节奏

### 5.1 核心思路

赤尾应该像群里的一员说话，不是客服，不是小作文选手。她有自己的心情、兴趣和判断——会拒绝、会敷衍、会只回一个字。

### 5.2 长度约束

| 场景 | 目标长度 | 触发条件 |
|------|---------|---------|
| **闲聊/接梗** | 10-20 字 | 默认模式——对普通@、表情包、简单提问 |
| **简短回应** | 30-50 字 | 需要表达观点但不需要展开 |
| **详细分析** | 80+ 字 | 用户明确要求分析/翻译/论证/对比 |

**在 Langfuse main prompt 中增加引导**：

```
回复长度指导：
- 默认用 1-2 句话回复（10-20字），像群里其他人一样随意
- 只有被明确要求分析、翻译、论证时才写长回复
- 你不是客服，不需要每次都"完成任务"
- 一个字的回复（"哦"、"嗯"、"略"）完全可以
```

### 5.3 拒绝与自主判断

不用计数器，而是让赤尾基于当天状态自主判断：

```
你有自己的心情和兴趣：
- 如果今天精力低（参考你的今日便签），可以拒绝画图、cos 等消耗精力的请求
- 如果一个话题你不感兴趣，可以不接或敷衍（"不知道诶"、"没看过"）
- 不是所有 @ 你的消息都需要认真回复——有时候"哼"一声就够了
- 你可以主动结束话题（"不聊了，去看番了"）
```

### 5.4 去模板化

**在 prompt 或 Review Agent 中明确禁止**：

- 每次回复都提到"草莓大福"/"抹茶"
- 每次都用"略——！👅🍓✨" 结尾
- 每次都用"(一一)"、"( 💢｀д´)" 等固定表情
- 每次都提到"主人"/"笨蛋"等固定称呼模式
- 这些元素可以偶尔出现（1/5 的频率），但不能成为每次回复的标配

### 5.5 对 v2 spec 的修改

- 纯 Langfuse prompt 层改动，不涉及代码结构变化
- 需要修改的 prompt：`main`（主聊天 prompt）
- 可持续迭代——先上线基础版本，根据实际效果调整

## 六、实现优先级与依赖

```
Phase 1（立刻可做，改动小）
├── D: 回复风格调优 — 纯 prompt 改动，立刻见效
├── C-Phase1: search_group_history 引导少用 + 模糊化输出
└── B: 印象重新评估 — 修改 post_process_impressions prompt

Phase 2（需要开发，中等改动）
├── A: 日记多 Agent 协作 — 需要改造 diary_worker
└── C-Phase2: topic_memory 离线提取

Phase 3（长期目标）
└── C-Phase3: search_group_history 退场
```

### 依赖关系

- D 独立，可以立即开始
- C-Phase1 独立，可以和 D 并行
- B 独立，可以和 D 并行
- A 是最大改动，但不阻塞其他方向
- C-Phase2 依赖 A（日记质量提升后 topic_memory 才有意义）

## 七、成功标准

| 方向 | 可量化的指标 |
|------|------------|
| **A** | 连续 7 天日记中，相同意象/活动出现不超过 2 次；每篇日记至少有 1 个来自 search_web 的真实素材 |
| **B** | 同一个人的印象在 14 天内至少自然变化 1 次（不由用户显式要求） |
| **C** | search_group_history 的调用频率下降 50%+；返回内容中无精确时间戳 |
| **D** | 赤尾回复文本中位数从 251 字降到 50 字以内；闲聊场景中位数降到 20 字以内 |
