> 状态：**草案（产品形态第二版，技术方案待讨论）**
> 分支：`feat/context-decline`

## 一、当前系统的问题

### 1. Context 注入信息重复

同一事件被多条管道加工后**同时注入 prompt**：

| 管道 | 产出 | 注入位置 |
|------|------|----------|
| afterthought | conversation fragment（~500 字内心独白） | `inner_context` |
| relationship extraction | core_facts + impression | `inner_context` |
| Life Engine | current_state + mood | `inner_context` |
| voice generator | reply_style | `voice_content` |

四者信息高度重叠，但都独立注入，占用 ~1100 tokens 传达 ~200 tokens 的增量信息。

### 2. 分层不统一

现有系统已经有"层"的概念，但每层各自独立产出、没有统一的状态机：

| 层 | 现状 | 问题 |
|----|------|------|
| 聊天消息（秒-分钟） | 直接注入 | ✓ |
| Life State（当下） | 直接注入 | ✓ |
| Glimpse（当日观察） | 产出后只喂给 Life Engine 和 daily dream | 孤儿层 |
| Conversation fragment（短期） | 直接注入 + 喂给下游多个管道 | 既是成品又是原材料，重复 10+ 次 |
| Daily fragment（中期） | 时间粒度压缩，喂给 schedule | 按时间而非按语义压缩 |
| Weekly fragment（长期） | 时间粒度压缩 | 几乎不被消费 |
| relationship_memory（长期） | 直接注入 | ✓ 质量最高 |

没有"一条记忆在不同层只是状态不同"的统一模型。

### 3. 长期记忆遗忘过度

```
373 条 conversation（~210k 字）
  → 25 条 daily（~28k 字）     压缩率 7.5:1（按时间机械压缩）
  → 3 条 weekly（~2.5k 字）    压缩率 84:1
```

压缩不管事情是否重要，也不区分是否已被关系记忆消化。

### 4. Recall 形同虚设

- 实现：PG full-text search（`simple` 字典）= 纯关键字匹配
- `relationship_memory_v2` 不在搜索范围
- 模型几乎不触发这个工具

### 5. Cross-chat 硬编码

`CROSS_CHAT_GROUP_IDS` 写死单个群 ID，其他群的跨群互动完全被忽略。

### 6. 数据现状（2026-04-16 快照）

| 数据源 | 记录数 | 平均长度 | 质量 |
|--------|--------|----------|------|
| conversation fragment | 373 | 564 字 | 内容好但高度重复（同一事件 10+ 变体） |
| glimpse fragment | 289 | 113 字 | 孤儿，信息密度低 |
| daily fragment | 25 | 1106 字 | 只覆盖 ~60% 天数 |
| weekly fragment | 3 | 827 字 | 只跑过一次 |
| relationship_memory_v2 | 617 | facts 89 字 / impression 134 字 | **质量最高**，具体行为+情感层 |
| life_engine_state | 1120 | ~100 字 | 同描述复用 35 次 |

## 二、产品形态

### 设计原则

**赤尾是一个人，记忆应该像人的记忆**。不用工程思维解决不确定性问题——该由 LLM 判断的交给 LLM，外部只提供辅助参考、不立硬规则。

### 1. 记忆的触发形态（三种组合）

- **场景自动注入**：特定场景自动唤起相关记忆（trigger_user 到场 → 关于 Ta 的抽象记忆；话题被提到 → 相关偏好/群氛围）。不依赖工具调用，像人见到熟人自然就想起 Ta 的事。
- **被动召回**：她觉得需要更多信息 → 调用 recall 工具按需加载更多记忆
- **主动回忆**：在对话中自发想起（"诶我想起来上次..."）

### 2. 记忆的内容类型

| 类型 | 形态偏向 | 示例 |
|------|----------|------|
| A 关于人 | 抽象 + 近期事实 | "他是工程师" / "昨天刚换工作" |
| B 关于事件 | 事实起步，淡化或抽象化 | "浩任艾特我三次" / "浩任老是烦我" |
| C 关于话题/偏好 | 抽象 | "我讨厌辣的" |
| D 关于群组 | 抽象 | "ka 群平时很吵" |
| E 关于自我 | 抽象 | "我最近变温柔了" |
| F 关于承诺/约定 | 事实 | "答应过 A 哥一起看电影" |
| G 共同回忆 | 事实/抽象都有 | "我和 A 哥一起吵过架" |

### 3. 两层记忆架构

```
┌──────────────────────────────────────────────────────┐
│  短期层（fresh，还没挂进 graph 的）                  │
│                                                       │
│  · 聊天消息（即时上下文）                             │
│  · Life State（当下活动/心情）                        │
│  · 新鲜事实碎片（最近几小时、reviewer 还没处理的）    │
│    - 与当前场景相关的（同 chat_id / 同 trigger_user） │
│      → 短期注入                                       │
│    - 不相关的 → 等 reviewer 整理                      │
└──────────────────────────────────────────────────────┘
                     │
                     │  reviewer 处理
                     ↓
┌──────────────────────────────────────────────────────┐
│  长期层（已挂进 graph 的）                           │
│                                                       │
│     抽象记忆（7 类统一存储）                          │
│        ↕  支撑边（graph）                             │
│     事实碎片（清晰 / 模糊 / 遗忘）                    │
│                                                       │
│  注入规则：                                           │
│    默认只加载场景触发的相关抽象（带支撑事实）         │
│  Recall 可以取到：                                    │
│    其他抽象 + 对应事实                                │
│    还没被遗忘的孤儿事实                               │
└──────────────────────────────────────────────────────┘
```

**关键性质**：
- 抽象记忆"骑"在事实记忆之上，通过 graph 边连接
- 事实被清除后，抽象仍然存在（如果还有其他事实支撑或已被充分强化）
- 孤儿事实（没被任何抽象支撑的）淡化更快——没被认知线索抓住的琐事本来就该忘
- 没有独立的"语义搜索事实层"——recall 是在 graph 上走边 + 遍历可记得的记忆池，不是关键字 / 纯语义搜索遗失的事实

### 4. 事实记忆的生命周期

```
产出（afterthought）
   ↓
短期层（新鲜）──→ 相关场景时可被短期注入
   ↓
reviewer 处理
   ├─→ 被挂到某抽象下（进入长期层，状态：清晰）
   └─→ 判定为琐事 → 快速淡化 / 直接清除
   ↓
reviewer 定期 review 已挂长期层的事实
   ├─→ 新事实进来强化抽象 → 旧事实可淡化
   ├─→ 情绪/关系/新奇性高 → 保留清晰
   ├─→ 长期没被引用 → 模糊化
   └─→ 被 recall 或对话引用 → 反向变清晰
   ↓
继续模糊 → 遗忘（recall 也搜不到）
```

**淡化的实现**：
- 档位概念存在但不预设太清楚——大致是"清晰 / 模糊 / 遗忘"但不严格离散
- 档位跃迁时内容重写：清晰 → 模糊时，reviewer 把内容改写得更抽象/少细节（"昨天和张三聊了 K8s 网络策略" → "前几天聊过一些技术话题"）
- 可逆：被 recall 或对话引用过的模糊记忆，下次 review 可以重新变清晰（但"清晰化"不是回到原文，是基于当下相关信息重构）
- 判断由 LLM 做：reviewer 看事实 + 外部辅助信号（上次访问时间、情绪强度线索、支撑抽象的数量等）后综合决定，没有硬规则

### 5. 抽象记忆的生成路径

**两条路径**：

- **对话中即时抽象**（赤尾自己）
  - 对话中意识到"我好像发现了关于 Ta / 这个话题 / 我自己 的新认知"
  - 即时产出一条抽象记忆（机制待讨论）
  - 适合"当场顿悟"

- **反思生成**（reviewer）
  - 离线 LLM 扫描最近的事实记忆
  - 发现 pattern → 产出新抽象，或更新已有抽象
  - 适合"需要时间才能看出来的规律"

### 6. 大脑 Reviewer

离线 LLM，扮演赤尾的**潜意识**。

**身份**：混合视角
- **全知视角**做决策：这条该淡化吗？该清除吗？要合并吗？
- **第一人称**生成内容：新抽象的文字用赤尾的口吻

**操作优先级**：

| 级别 | 操作 |
|------|------|
| **P0** | 更新抽象 · 新事实进来强化/修正已有抽象 |
| **P0** | 标记清晰度 · 事实档位跃迁（含内容改写） |
| **P0** | 调整支撑边 · 抽象更新时同步维护 graph 边 |
| **P1** | 合并相似事实 · 解决 fragment 重复生成的痛点 |
| **P1** | 创建新抽象 · 从 pattern 产出新抽象 |
| **P1** | 清除事实 · 仅对"已被充分消化 + 长期无引用"的事实生效 |
| **P2** | 检测矛盾 · 冲突触发"认知失调"（单独立项） |

### 7. Recall 的定位

**不是**"搜索遗失的记忆"，**是**"能记得的事太多了，默认只加载最相关的；recall 是主动去触达其他还记得但没加载进来的"。

- 搜索范围 = 所有没被遗忘的记忆（清晰 + 模糊）
- 实现形式不重要（graph 遍历 / 语义匹配 / 混合都行）
- 返回内容包含事实 + 其挂载的抽象记忆（上下文更完整）
- 赤尾看到的是"当前档位下的内容"——模糊档位就是模糊文字，自然表达，不需要"我记不清了"这种 meta 话术

## 三、现有管道的调整

| 管道/数据 | v3（现状） | v4 调整 | 说明 |
|-----------|-----------|---------|------|
| conversation_messages | 直接注入 | 保持不变 | 即时上下文 |
| Life Engine state | 直接注入 | 保持不变 | 当下活动/心情 |
| conversation fragment | 直接注入 + 喂下游多个管道 | **只产出、不直接注入**；作为事实记忆进入两层架构 | 短期新鲜时可按相关性短期注入；reviewer 整理后进长期层 |
| glimpse fragment | 只喂 Life Engine / daily dream | **并入事实碎片层**（打 tag 区分来源：观察 vs 对话） | 孤儿升级为一等公民，由 reviewer 统一处理 |
| daily fragment | 每日 cron 产出 | **废弃** | reviewer 直接消化事实进抽象，不需要按时间粒度压缩的中间层 |
| weekly fragment | 每周 cron 产出 | **废弃** | 同上 |
| relationship_memory_v2 | 直接注入（per-user） | **扩展为统一抽象记忆层**，覆盖 7 类 | per-user 只是 A 类；其他类型共用同一张表 |
| voice / reply_style | 直接注入 | 保持（或后续单独评估） | 当前不在此次 scope |
| cross-chat | 硬编码单群 | **去硬编码**，按 trigger_user 在所有群互动过滤 | 解决人格分裂问题的设计初衷保留 |

## 四、目标态下的产出/消费矩阵

| 数据 | 产出者 | 消费者（加粗 = 直接注入 prompt） |
|------|--------|-----------------|
| 聊天消息 | 用户/赤尾对话 | **inner chat history**（原本就有） |
| Life State | Life Engine | **inner_context** / voice |
| 新鲜事实碎片 | afterthought | reviewer / **短期相关时注入 inner_context** |
| 长期事实碎片 | afterthought 经 reviewer 处理 | reviewer / recall（按需） |
| 抽象记忆 | 对话中即时抽象 + reviewer | **场景触发时注入 inner_context** / recall（按需） |
| reply_style | voice generator | **prompt voice_content** |
| cross-chat | 原始对话记录 | **inner_context**（按 trigger_user 相关性过滤） |

**关键原则**：每条数据只在**加粗**的位置被注入，其他管道只消费不注入。

## 五、待讨论的产品细节

1. **新鲜事实短期注入的相关性规则**：同 chat_id / 同 trigger_user / 主题相关，哪些是必要条件？时间窗口多长？
2. **场景触发的触发规则**：什么场景信号（用户/话题/群/时间）触发哪类抽象记忆的注入？
3. **In-conversation abstraction 的实现机制**：tool call（明确决策）/ 后置提取（自然流露）/ 混合
4. **Proactive recall 的时机**：赤尾在什么情况下自发调用 recall
5. **冲突处理**（F 操作）：新事实和旧抽象矛盾时的反应
6. **Graph 边的丰富度**：只有 abstract↔factual / 还要 factual↔factual（因果/时序）/ 还要 abstract↔abstract（层级）
7. **Reviewer 运行频率**：每日 / 每几小时 / 事件触发 / 混合
8. **Reviewer 对新鲜事实的处理延迟**：新鲜事实多快会被 reviewer 摸一次？这影响"从新鲜层到长期层"的滞后
9. **历史数据迁移**：现有 617 条 relationship_memory_v2 + 373 条 conversation fragment + 289 条 glimpse 怎么平滑过渡
10. **废弃 daily/weekly 对现有下游的影响**：schedule 生成是否依赖 daily？如依赖，替代方案是什么？
