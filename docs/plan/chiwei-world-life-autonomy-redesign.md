# 赤尾 world / life 自主化重设计 —— 目标架构

把 world / life 从"world 当导演、替角色移动和裁决、用 event 命令角色反应"的范式,重做成"world 只推演世界并把感知送达角色、角色完全自主"的范式。这是赤尾设计原则在引擎架构上的彻底贯彻。

## 核心原则

**不存在任何直接干涉或告知角色行为的外部机制。角色完全自主——何时醒、做什么、去哪,都她自己定。** world 只提供世界感知(输入),角色作为 agent 自主产出行为。不确定性是她像人的来源、不是要消除的 bug,所以系统从不替她拍板:不替她选择做什么、不裁决她想不想、不命令她反应。

注意这条原则的边界:它管的是**角色的决定**,不是**世界的客观事实**。world 不替角色选目标,但 world 仍是客观物理的权威——谁在哪、唯一资源谁占着、一个动作物理上能不能成,这些客观裁判仍归 world。退位退的是"替角色做决定",不是"维护世界一致性"。

## 目标架构

### world engine:世界推演者 + 双通道感官 + 物理裁判(不替角色决定)

world 仍是会思考的 agent,但它想的是**世界长什么样、谁此刻能感知到什么**,绝不想角色该干嘛。它干这几件事:

- **推演世界客观状态**:综合角色的动作 + 环境演化(时间在走、天气变化),推演"此刻世界是什么样"。
- **把感知送达角色(两条通道)**:
  - 在场通道——物理场景的客观动静,按"谁在这个场景"分发。绫奈在家就 notice 她"厨房在做饭",她在外面就感知不到家里。感知尺度就是"在不在场",不做气味 / 声音传多远那种物理推演。
  - 订阅通道(手机)——外部 / 数字事件(番剧更新、新闻)接**真实数据源**,按"谁订阅了"分发给对应角色。
- **支持角色 ask world**:角色主动问"此刻周遭 / 我手机什么情况",world 答她够得着的部分(受她的在场 / 订阅限制,不是全知)。
- **物理裁判**:角色发起一个动作(我要去厨房、我要用浴室),world 裁判它在客观世界里的结果——能不能到、唯一资源被占了就得等、移动需不需要时间——并据此**权威更新 presence**。presence 不是"谁动谁写",它是 world 维护的客观事实。world 裁的是物理可行性,不是角色的目标选择。
- **自己醒来推演环境**:天黑、下雨这类环境自发变化,由 world 自调度醒来推演,自己定下次多久再看一眼,不被固定心跳强制拍醒。

**感知输出契约(钉死,不只靠 prompt):** world 送达角色的一切(在场通知、ask world 的返回)必须是**客观事实的呈现**——谁、在哪、什么客观可感的动静,按在场 / 订阅过滤可见性,**禁止夹带评价、建议、"你该做 X"的指令**。导演式干预最容易藏在通知的措辞和 ask world 的返回值里,所以这层要靠交互契约(形态约束 + 可见性过滤 + 禁止项)钉死,而非寄望 prompt 自觉。

### life engine:完全自主的角色

- **三种驱动**:被 world 通知(世界变了 / 手机来消息)、主动 ask world(自己看一眼周遭 / 查手机)、自排下次醒(她自己的生活节奏,闹钟是它的一个用法)。
- **自主行动**:自己决定醒不醒、做什么、去哪。移动是"发起一个移动动作(我要去厨房)"汇给 world,由 world 裁判物理结果——**她选目标,world 落实**,presence 的权威在 world。
- **角色间交流**:对话的**内容**在 life engine 内部直连(不每句话绕 world);但"两人在某场景交流"这件事作为一条**客观在场动静仍经 world**——这样第三人进场能感知到"有人在说话"、世界一致性不破。需要定义会话边界:谁在场能参与 / 听见、第三人进场怎么接入、对话何时自然结束。防无限对话靠 prompt 约束 + 会话边界,不上计数器。

### 一句话的循环

角色自主行动 → 动作汇回 world → world 裁判物理结果 + 推演世界新样子 + 谁该感知到 → 通知对应角色(或角色主动 ask world)→ 角色感知、自主下一步行动 → ……  world 是循环里"推演世界、裁判物理、分发感知"的枢纽,角色是循环里自主的人。

## 感知与信息的责任划分

- 物理在场感知(做饭、下雨、有人进屋)→ world 在场分发
- 外部订阅(番剧更新、新闻,接真实数据源)→ world 订阅分发(手机)
- 闹钟(角色自己可预知时刻的定时)→ life 自排,world 不插手
- 真人飞书聊天 → 待定(见开放问题)

划分口径:角色能自己预知时刻的(闹钟)归 life 自排;角色预知不了、必须外部在事件发生时推给她的(在场动静、订阅更新)归 world。

## 可靠性与意愿分离

角色完全自主 + 自排,理论上可能"大家都选择不动→世界静止"。这不能用固定心跳强行"叫她们动"——那就是隐性外部推动、违反核心原则。系统只需要一个**中性的 liveness / watchdog**,它只兜**机械失败**:投递失败没送到的、自排丢失没醒的、信箱有未读却没被唤醒的——把该送的补送、该醒的补醒。它**绝不凭空生成角色行为、不强推她们动**。可靠性(机制故障)和意愿(角色想不想)严格分开:watchdog 修的是"消息没送到",不是"她不想动"。

## 上下文建设

- **每个主体当天一条意识流 session**:角色 `{lane}:{角色}:{日期}`、world 一条,按天切。这条 session 就是该主体当天"活着"的会话实体——你看到的连续 langfuse session 背后真有连续上下文。
- **三层各就各位**:
  - system prompt = 纯静态身份:性格 / 世界观 / 规则 / 工具说明 + 恒定的角色设定(名字、persona_lite)。任何会变的——几点、上一刻状态——都不许进,否则前缀缓存复用全废、语义也错位。
  - transcript = 当天连续意识流:她一路的感知、行动、对话、状态变化都在流里自然延续。"上一刻什么状态"不再从 PG 现捞塞进 system,意识流里本就有。
  - 当前 USER = 这一轮新感知:几点、信箱里新的客观动静,每轮现拼一条进流。
- **状态恢复源**:删 prev_state 出 system prompt,但 LifeState(PG 持久硬事实)保留——transcript 在 Redis(24h、非 durable),丢失 / 冷启动时靠 PG 的 LifeState 恢复"她醒来记得自己之前在做什么",作为冷启时的一条感知喂进当前 USER,而**不是**常态塞回 system prompt。正常情况状态从意识流延续,LifeState 只在意识流断了时兜底。
- **跨天靠 memory 沉淀**:当天重要的经历 / 事实沉进长期记忆,第二天换一条新意识流、但她通过 memory 记得昨天要紧的事。像睡一觉——具体对话细节翻篇了,要紧的记得。
- **时间统一 CST**:喂给 agent 的所有时间显示成 CST、所有时间比较按真实时刻归一、存储一律 aware ISO。

## 与现状的差距

- **删**:`move_persona`(world 替角色移动并决定)、`raise_intent → world 裁决意图`链路、唤醒时夹带的"你该反应"指令、system prompt 里的动态变量(current_time / prev_state / prev_mood / prev_activity)。
- **改**:world 从"导演 agent(move / emit / 裁意图)"改成"推演者 + 双通道感官 + 物理裁判";presence 写入方仍是 world、但触发从"world 主动挪"改成"角色发起动作、world 裁物理结果";life 从"被 event 唤醒的无状态节点"改成"自驱循环、自己发起移动动作";唤醒语义从"命令她反应"改成"通知世界状态、她自主决定"。
- **加**:角色发起移动 / 动作的契约;受约束的感知输出契约;ask world;角色间交流的会话边界;中性 watchdog;订阅通道(手机 + 真实数据源);跨天 memory 沉淀。

## 分阶段

**阶段 0 — 地基(不动范式,可先做先验,风险低)。** 把时间和上下文两个地基理顺:时间统一 CST(喂 agent 全 CST、比较归一、存储 aware ISO);上下文三层归位(system prompt 清成纯静态、动态挪进 USER、当天状态靠 transcript 延续,LifeState 留作 transcript 丢失 / 冷启的状态恢复源)。验证:trace 里 world / life 的 system prompt 纯静态、不含时间 / 状态变量,所有时间 CST,同一 session 多轮续接、状态从意识流延续;杀掉 Redis 模拟 transcript 丢失,角色仍能从 PG 的 LifeState 恢复当天状态、不彻底失忆。

**阶段 1 —— 核心范式,拆三小步、一刀一验(防一次性全切翻车)。**
- **1A 去裁决 + 角色自移动 + 受限感知契约**:删 move_persona 与 raise_intent→裁决;角色发起移动动作、world 裁物理结果并权威更新 presence;world 通知 life 改成受契约约束的客观感知(可见性过滤 + 禁夹带指令)。验证:单角色被世界状态通知后自主移动自己、全程不经 world 裁决意图;world 侧不再出现替角色选择;并发移动 / 唯一资源占用由 world 正确仲裁。
- **1B 自排唤醒**:life / world 自排下次醒(吸收 [chiwei-world-self-schedule-cst-time-spec.md](chiwei-world-self-schedule-cst-time-spec.md) 的 self-schedule + sleep 动态上限;CST 部分已归阶段 0);中性 watchdog 只兜机械失败。验证:world / 角色按自排节奏醒、不被固定心跳强制;断网 / 漏投时 watchdog 能补、且不凭空造行为。
- **1C 同场景直连交流**:角色间对话内容走 life 直连,"交流在发生"作为客观动静经 world,定义会话边界。验证:两角色同场景能直接交流不绕 world;第三人进场能感知到"有人在说话";对话能自然收束、不无限刷屏。

**阶段 2 — 扩展(后续)。** 订阅通道(手机:番剧 / 新闻接真实数据源)+ ask world;跨天 memory 沉淀 + 意识流压缩 / 持久化加固。

## 开放问题

- **真人飞书聊天的归属**:之前 self-schedule spec 里设计成"聊天打断 world",在新范式下 world 不该被聊天直接驱动——聊天更像"手机来消息"送达角色 life。这条要按新范式重新审视。
- **订阅真实数据源怎么接**:番剧 / 新闻源的具体形态、接入方式。
- **跨天 memory 沉淀**:怎么和现有 memory 系统(core_fact / 关系等)对接、边界在哪、什么时机沉淀。
- **意识流持久化**:按天 + 24h Redis 缓存够不够,要不要 PG 落盘加固(丢了 = 当天失忆,目前靠 LifeState 兜部分状态、但意识流本身仍会丢)。
