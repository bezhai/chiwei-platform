# life 自主发消息:收拢旧旁路,打通统一发送

> 2026-06-15 起草,codex T1 两轮 + "当面/手机两模态"对齐后修订。承接 `chiwei-more-human-directions.md` 里"主动给真人发第一条消息"的方向,这一刀只做"隔手机发消息"这一种,当面聊天与延迟引用回复等另立。

## Problem

旧有一条独立旁路:定时去判断某个会话里有没有赤尾感兴趣的话题,有就让她回那条消息(`PROACTIVE_USER_ID` 那套)。它现在已是死代码,设计上也拧巴——把"要不要说话"外包给一个确定性判断器,而且只能挂在一条入站消息上做反应,做不到赤尾凭自己的生活节奏主动找人说话。与此同时,life 现有的 `chat` 工具收件人写死成三姐妹,既发不到真人,也意味着将来想给任何新出现的人发消息都得改代码。

## Goal

删掉旧 proactive 旁路死代码。给 life 一个主动发消息能力:她在生活流里自主决定发不发、发给谁、发什么;报个名字模糊查到候选(全局 uid + 简介),挑一个后发出去。**核心价值是能扩展到任意"其他人"——新出现的人进了名单她就够得着、不用改代码;若只为现在这几个对象,写死就够了,根本不必做成查询。** uid 是稳定的身份句柄,一个 resolver 负责把 uid 解析成"此刻该怎么投递"。这一刀只接飞书。

## 两种交流模态(这一刀的关键区分)

赤尾世界里的"交流"分两种,模型现在常把它们混为一谈(老把飞书群聊、私聊当成当面),必须分清:

- **当面说话**:在同一个场景里面对面。只有"我(bezhai)"和三姐妹之间支持——三姐妹彼此当面,我也能跟三姐妹当面。但我是真人、不在她的虚构世界里,所以她"当面对我说"落地会退成飞书私聊给我发。
- **隔手机发消息**:不在一起时发消息,所有人都支持——三姐妹不在一起时互发、给我和其他真人发飞书。

两个推论:其一,给真人的送达统一都走飞书私聊(这是当前的送达策略,不是"每个真人都必然可私聊投递"的身份假设——能不能投递由 resolver 说了算);"当面还是远程"对真人不是两条送达线,区别只在 world 的世界设定里我这会儿在不在她身边。其二,发消息这件事上"我"就是普通真人、不特殊,我唯一特殊的地方是"能跟三姐妹当面"。

这一刀只做"隔手机发消息"这一模态。当面聊天机制本身、以及 world 怎么安排真人何时入场("我什么时候能跟赤尾当面")拎出去另立——后者是 world 的职责,且是个独立的大题。

## Non-goals

- 当面聊天机制本身;world 安排真人何时入场("我何时能当面")——另立。
- 隔天"引用回复"一条历史消息、让她感知"自己没参与的群聊"、"真去读完小说再回来接话"——终极能力,另立。
- NPC 发消息——先搁置。
- 多渠道(qq 等)实际接入——uid 是渠道无关的身份句柄、天然留了扩展位,但这一刀不写多渠道代码。

## "proactive" 一词的两个含义(动手前必须分清)

代码里 `proactive` 同时指两个不相干的东西,混淆会导致误删:

- **(a) agent-service 侧的旧外部判断器旁路**:`PROACTIVE_USER_ID`、`find_proactive_messages_in_chat`、`insert_proactive_message`、`app/chat/context.py` 里的 `is_proactive` 分支。这是要删的死代码。
- **(b) channel-server `chat-response-worker` 里的 `is_proactive` 出站分支**:这是"主动发到一个会话"的出站模式,要复用、**绝不能删**。

任务里按具体符号精确区分,不要凭 `proactive` 这个词一锅端。

## Key design decisions

1. **全局 uid 是稳定的身份句柄**,用 typed uid(如 `persona:akao` / `user:<common_user_id>`),不新建 contact 表。简介从现有 persona / common_user 资料取。"我"就是 user 里的一个,不特殊对待。
2. **身份与投递目标分两层**。uid 回答"这是谁",resolver 回答"此刻怎么把消息送到 ta"。真人私聊投递的合同要闭合,不能靠伪 id:优先用已有的 p2p 会话地址;能不能用对方 open_id 主动发现/新建私聊、由哪个 bot 身份发送,要查实并定策略;群聊里见过的真人不必然能私聊投递。查不到或不可投递时 resolver fail-loud(明确报"发不了"),不静默糊弄。
3. **名字模糊查只返回候选(uid + 简介)交给 life 选**,绝不排序、不按亲密度/活跃度/兴趣筛、不自动取第一个。否则会在"身份解析"里悄悄复活一个替她决定跟谁说话的确定性模块。选谁是 life 的决定。
4. **模态(当面/手机)也是 life 自己选的**,不是代码替她判断。代码不根据"谁在场/亲密度/最近聊过"去替她决定该当面还是手机、发给谁、发不发——只按她明确选的 uid 和模态做机械投递。
5. **发送分发,且收件人侧能区分模态**。这一刀新增"隔空发":三姐妹不在一起时手机互发、真人走飞书私聊;三姐妹之间现有的当面 speech 信箱保持不动。关键:手机消息和当面 speech 落到收件人那里必须可区分(不同事件类型/不同展示),否则会在姐妹之间又制造一遍"当面还是手机"的混淆,task 5 也就白做。
6. **错误显式喂回 life**:无候选、重名未决、uid 不可投递、查不到会话地址、bot 无权限、出站失败,一律作为工具错误反馈给 life,让她自己处置(换个人、重试、或算了),不静默降级、不代她另找目标。
7. **上下文标清两个正交维度,不是一个全局标签**:一是"物理在场"(她此刻在什么场景、跟谁面对面),二是"通信介质"(她正通过飞书群聊 / 飞书私聊 / 未来 qq 跟谁打字)。一个 life 轮里物理场景、手机消息、信箱事件可能并存,把它们压成一个"当前场合"字段正是"把飞书当当面"的根。两维度各自作为可追踪的输入标清。
8. **全程 agent 自主**:发不发、发给谁、发什么、当面还是手机,全由 life 的 prompt 自然涌现,代码只提供"查名字、按 uid 发"两只手,不回任何替她判断的模块。删旧旁路正是为此,新实现不得变相复活它。

## Caller coverage

- life 的 `chat` 工具(`app/nodes/life_tools.py` 的 `build_life_tools`):现在收件人写死 `SISTERS_CONTACTS`,要改成按 uid 发、能覆盖真人;同时理清它现有"当面/发消息混用"的语义,把当面 speech 与新增手机发区分开。
- `SISTERS_CONTACTS` 常量的所有引用方。
- 待删死代码(上节 a 类):动手前 grep 全部引用,确认确为死代码、删后已有测试套不挂;同时确认没有误伤 b 类出站分支。
- 出站现状:`chat_node` 现有 emit `ChatResponseSegment`(被动回复)与 worker 的 `is_proactive` 分支——核实它是否依赖真实来源消息反查渠道地址,主动发没有来源消息时这条路径要怎么拿到可投递的会话地址与 bot 身份。
- 上下文拼装处(life 与 chat 的 context):要加"物理在场"和"通信介质"两个维度的标注,核对两边都覆盖。
- 真人投递标识来源:`common_user` / `common_conversation`,确认能查到 p2p 会话地址、open_id、可用的 bot 身份。

## Data & deployment impact

- 不新建 contact 表(typed uid + resolver);只有在简介或投递地址实在查不到时,才考虑补最小存储。
- 真人出站的核心未知点:worker 现路径是否依赖真实来源消息反查渠道 id?若依赖,resolver 必须直接给出可投递的私聊会话地址和发送 bot,不能靠伪 id;查不到就 fail-loud。
- 上下文两维度标注是改 prompt 输入层,无 schema 变更。删死代码本身也无 schema 变更。
- 部署:agent-service;若出站字段需要改动,则同步 channel-server(chat-response-worker 同镜像)。coe 飞书 e2e 需绑 dev bot。

## Tasks

1. **收拢旧旁路**。目标:删掉 agent-service 侧 proactive 外部判断器死代码(a 类),明确不动 channel-server worker 的 `is_proactive` 出站分支(b 类)。产出:相关函数、常量、context 分支全部移除,调用方零残留。验收:grep a 类旧符号为零、worker 出站路径完好、已有测试套全绿。
2. **typed uid 与模糊查询 resolver**。目标:按名字查到候选 uid + 简介,并能把 uid 解析成投递目标。产出:覆盖三姐妹与真人的轻量 resolver(typed uid,不建表,"我"即一个 user),姐妹 uid 解析到信箱、真人 uid 解析到飞书私聊地址 + 发送 bot,查不到则 fail-loud。验收:一个模糊名字返回候选 uid + 简介;给定姐妹 uid 解析出信箱目标,给定真人 uid 要么解析出可投递的私聊地址、要么明确报不可投递。
3. **life 手机发送工具**。目标:她在 life 轮里自主按 uid 隔空发消息,发送层按对象分发,错误喂回,且收件人能区分手机消息与当面话。产出:`chat` 工具改成按 uid 发(三姐妹不在一起时手机互发、真人走飞书私聊),手机消息与当面 speech 在收件人侧用不同事件类型/展示区分,不可投递等错误作为 tool error 反馈。验收:一轮 life 里她给姐妹手机发一条、收件人上下文能看出"这是手机消息"而非当面;给真人发进出站队列;给一个不可投递 uid 时她收到错误而非静默失败。
4. **真人出站到飞书私聊打通**。目标:主动发的消息真送到飞书真人私聊。产出:用 resolver 给的真实私聊地址与 bot,复用 worker 既有 `is_proactive` 出站路径送达(不靠伪 id)。验收:coe 绑 dev bot,真人在飞书私聊侧收到她主动发来的一条消息。
5. **上下文标清两维度**。目标:治"把飞书群聊/私聊当成当面"的混淆。产出:在 life 与 chat 的上下文里分别标明"物理在场"和"通信介质"(飞书群聊 / 飞书私聊 / 当面 / 未来 qq),不压成单一标签。验收:trace 里看到两个维度的结构化标注都覆盖到,再加少量固定回归样例通过(不以"模型不再说错"作为唯一准入)。

依赖:任务 2 → 3 → 4 串行;任务 1、5 可独立先做。
