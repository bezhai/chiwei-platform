# 备忘录 & 日程:赤尾手机里的第一个 app

> 2026-06-13。"手机"概念落地的第一个 app。方向背景见 `chiwei-more-human-directions.md`。
> 这份 spec 重在"它怎么活"——怎么嵌入、怎么记、记什么、过期咋办、谁消费;技术实现(表结构、字段、函数)留到真动手时再生成。

## 现在的问题

她"惦记的事"和"打算几点干嘛"都没地方放——上一轮想的事下一轮就蒸发,说"在玩手机"也落不下任何东西。旧版有过一个 note 工具,但早就删干净了(连底层表带查询一起拔了),而且它那套(挂在 chat 上、堆成一堆没人清的待办)也不是我们要的。她现在的"自排醒来"只能排"下一轮大概啥时候醒"这么一条笼统的,挂不住"好几件带内容、到点提醒我"的真日程。

## 目标

给她一个随身的小本子:在 life 里她自己记没时间的事(备忘录)、排有时间的事(日程);到点提醒她、她自己处理;聊天时这些事自然就在她脑子里;过期的不会堆成一摊僵尸待办。

## 不做什么

- **不负责消除"看完 / 做完一眼假"**——那靠后面"真做事 = 一个 agent 工具"那条主线(比如真去读的阅读 app)。备忘录 & 日程只是底座:她能记下"想看 X""看完了 Y",但"看完得到的真观后感"的真来源不在这份 spec 里。这条写明,免得验收时误以为这一块就能让她不再编。
- 不做手机的其他 app(相册 / 闹钟 / 阅读)——那些后面照这个样子各自做。
- 不在 chat 侧单独存一份。chat 是 life 的快照,本子在 life。
- 不从对话 / 意识流里自动抽取待办——记不记、记什么,她自己定。
- 不做优先级 / 标签 / 分类那套结构化——人记备忘不填表。
- 不做死课表、不到点强制执行。
- world 不读这个本子(那是她私人的内心)。

## 关键设计决策

**怎么嵌入** — 它是 life 里的一个能力,不是独立服务:在她的工具集里加一组她能用的动作(记一笔、排个时间、翻本子、改、划掉),再加一段"她随身本子里还没了结的事"进她每轮的输入。chat 作为 life 快照带上(工程上要显式接,见下)。这是"手机"的第一个 app,后面相册 / 闹钟照这个长。

**怎么记、记什么** — 她自己在 life 循环里决定记什么,系统不替她从对话里抽(选她自主、不选自动提取:自动抽会记一堆她其实没在意的,而且是工程脑)。内容就是她用大白话写的一句话(选自然语言、不选结构化字段:人记备忘不会去填标题 / 优先级 / 标签)。两类条目,差别只在挂没挂时间:没时间的是备忘录(想做的事、惦记的人、突然的念头),有时间的是日程(到点要做 / 被提醒)。**一个本子、两种条目,不拆成两个**——一条事哪天加个时间,就从备忘录变成日程,拆开就得来回搬。

**日程到点怎么调度**(现状的真缺口,别想轻了) — 她现在的自排醒来只有一个"下次什么时候醒"(单条、一排就覆盖上一条、只带时间不带"为什么醒"),撑不起"好几条各自独立的日程、能改期能取消、还可能同时到点"。这块的调度契约要在动手时定死,至少答清楚:每条日程是各挂各的提醒、还是只盯最近一条到点了再续排下一条;改了期 / 划掉之后原来那个提醒怎么作废、不误触发;同一时刻撞上好几条怎么收。我倾向每条日程各挂各的(像她行动那样各投各的),但具体落地留到第三块动手时定。**不动她现有"自排下一轮节奏"那套语义,日程是在它旁边新加的一路。**

**日程过了咋办**(旧 note 踩过的坑) — 三层,全是她自己的判断、没有任何定时器:
1. **到点**:复用现有"到点把她叫醒"的地基,把那条日程递到她面前,她当场自己处理——去做了就划掉、想再拖会儿就改时间、不想做了就划掉。系统不替她判完成、不强制执行。
2. **没被处理的过期条目**:不搞"到期自动删"(那会删掉她其实还惦记的,而且是死规则),而是在她**睡前回顾**那一班给她一个固定的"翻本子"时机——做过的 / 过时的 / 不想做的她自己清掉,还惦记的留着或重新排个时间。
3. **平时进她脑子的,是她自己还没了结的条目**——"还活着"指的是她自己没标"做了 / 不做了"的那些,**不是代码按年龄、条数、过期时间去筛**(那就成了代码替她决定忘掉什么、违宪)。这一步只是只读地挑着显示、不改状态、不删任何东西;被省下没显示的旧条目,她翻本子随时还能看到。条数别涨到失控,靠的是她睡前自己清,不是代码截断。旧 note 的病就是一堆老待办永远常驻、一眼假——根上是它没有"她自己了结"这一层。

**谁去消费** —
1. **她自己每轮醒来(life)**:本子里还没了结的 / 今天要做的进她输入,她带着过日子,可能因此去做某件事、或想起某个人。
2. **她自己聊天时(chat 快照)**:本子里的事在她脑子里,她能自然聊到("我周末要陪我妹")。
3. **睡前回顾**:读本子帮她清陈年的,把做过的事沉淀进她当天的日记。
4. **到点唤醒地基**:日程到点把她叫醒、递给她。
5. world 不消费(守信息差)。

## 她对本子能用的工具

系统自动带的(lane / persona / 当前时刻)不算入参,下面只列她要填的。

1. **记一条** — 往本子写一件事。
   - 入参:内容(必填,一句大白话);提醒时间(可选,挂上 = 日程到点提醒、不挂 = 备忘录躺着)。她按自己输入里的"现在几点"算出时刻填。
   - 出参:这条的 id(以后改 / 划得指到它)+ 一句确认。
2. **改 / 划一条** — 动已有的某条。
   - 入参:条目 id(必填);改内容(可选)、改时间(可选,含给没时间的补时间 / 把时间撤了)、状态(可选:做了 / 不做了划掉)。
   - 出参:确认改了什么。
3. **翻本子** — 她想完整看一遍时(找条旧的、睡前清理)。
   - 入参:看还活着的(默认)/ 看全部(含做过、划掉的)。
   - 出参:条目列表,每条带 id、内容、时间(有的话)、状态(还惦记 / 到点了 / 做了 / 划了)。
   - 平时她不用主动翻——还没了结的条目每轮自动进她输入。但因为旧条目不常驻输入,这个"翻本子"是她拿回"看全部"能力的**必要出口**(找旧条目、睡前清理都靠它),不是可选。

都是 durable 工具(她记的真落库)。"记一条"的 id 从触发源派生、整轮重试不重复记(durable mutation 幂等、max_retries=1),实现时落实。

## 调用方覆盖

- **life 工具集**(`build_life_tools`):新增她的本子动作。纯新增,不改现有工具语义。
- **她每轮的唤醒输入**(life 醒来时拼给她的内容):新增"本子里还没了结的事"一段。
- **chat 对话上下文**(`inner_context`):概念上 chat 是 life 快照,但工程上 `inner_context` 现在是显式拼几段(life 状态 / 昨天页 / 关系页),本子**得显式接进去**才会出现在聊天里,不是自动就有。
- **睡前回顾**(life review):新增"翻本子、清理 / 沉淀"一件事。
- **现有"到点叫醒"信号**(self-wake / LifeWakeTick / 到点 gate):日程到点复用它把她叫醒。现在这个信号只带时间、不带"为什么醒",要补成到点时她知道是哪条日程——**只补这个,不动它现有的自排语义**。多条日程的调度契约见上方「日程到点怎么调度」。
- **world**:明确不读本子。

## 数据 & 部署影响

- 新增一处存她本子的 durable 地方(按 framework Data 建,遵守新 Data 三步检查:grep 保留列名 + 读 sibling Data + 端到端 insert 验证)。
- 不删不改现有表,纯新增。
- agent-service 单服务;部署会重启 world / life 循环(正常,部署前确认没有正在跑的批量任务)。
- 大概率有 prompt 改动(她怎么用本子、睡前怎么清),走 langfuse 泳道 label,先泳道后 production。

## 任务

1. **本子的底子 + 她记的能力**。目标:她能在 life 里记一笔(没时间)、排一条(带时间)、翻、改、划掉。产出:存本子的 durable 地方 + 她工具集里的本子动作。验收:在 coe 让她记几条、排一条带时间的,落库查得到、能改能划。
2. **进她脑子(她自己 + 聊天)**。目标:她还没了结的条目进她每轮输入,聊天时也在她脑子里。产出:她唤醒输入里的本子段 + 把本子显式接进 chat 的 inner_context。验收:她唤醒的 trace 里输入带着本子内容;chat 的 trace 里也能看到本子那一段被拼进了对话上下文、她聊天能自然提到本子里的事。
3. **日程到点提醒**。目标:日程到点,现有唤醒地基把她叫醒,她当场知道是哪条、自己处理。产出:到点唤醒携带日程信息 + 她处理日程的闭环。验收:在 coe 排一条近的日程,到点她被叫醒、输入里点出这条、她能处理掉;并覆盖这些边界——多条几乎同时到点、排了个已经过去的时间、到点时她正在睡觉、改期或划掉之后原来那个提醒不再误触发。
4. **睡前清本子 + 沉淀**。目标:睡前回顾翻本子,陈年的 / 做过的 / 不想做的她自己清,做过的事沉淀进日记。产出:回顾班里的翻本子环节。验收:跑一次回顾,过期 / 完成条目被她处理掉、不再常驻输入,做过的事进了她当天的页。
