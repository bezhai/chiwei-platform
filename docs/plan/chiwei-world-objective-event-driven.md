# 赤尾:world 纯客观事件驱动 + life 事件反应 范式

## 问题

现状(world-driven wake,commit 9e87164d)里,world 除了推演世界,还兼着判断"哪个角色停太久了、该不该叫醒谁",再发唤醒通知。这个判断会出错:life 一静止就不产生行动,world 下一轮 pull 到空、看世界"没动静",于是判断"没必要叫谁",越静越不叫——形成自锁。实测今早三姐妹卡死 2 小时 20 分就是这么来的。另外 life 自己还写 next_wake_at 当唤醒意愿,但它不可靠(设错 / 进程重启丢失就永久睡死),而且"她到点该醒"这种存活保障本不该压在她自己手上。

根因有两层:一是 world 把"要不要唤醒角色"当成一个会判错的主观决策;二是 life 的存活依赖她自己设的闹钟这个脆弱单点。

## 目标

用离散、稀疏的模型调用,模拟一个连续生活的真人:接触她的人感觉她是连续活着的,她后台确实在过一段真实、可追溯、不无趣的生活,成本控制在 100 美元/天以内,且不因为持续高频运行而让模型输出降质。

具体落到机制:world 退成纯客观世界推演者,只陈述客观世界发生了什么、绝不判断该叫醒谁;life 退成纯事件反应者,被客观事件激活才动、自己绝不设闹钟;她的存活由世界持续不断的客观事件流兜底,她的主动计划由"日程"承载。

## 不做什么

- 不给 life 加回任何形式的自设闹钟(next_wake_at / self-wake 整条删,不是改可靠)。
- 不让 world 判断"该唤醒谁"——收件人由客观在场决定,不由 world 主观挑。
- 不引入随机事件池 / 作息时刻表 / 状态机 / 阈值规则去制造世界的丰富或节奏——节奏和内容都由 agent 基于客观事实推演(赤尾宪法)。
- 不重写底层(状态持久化、dataflow 框架、飞书接入、记忆、外部源),基于当前分支 feat/life-web-access 改。

## 核心范式(关键设计决策)

**1. world = 纯客观世界推演者。** 它只做一件事:基于真实时间 + 拉来的真实外部信息(天气 / 时事 / 日历),推演客观世界此刻发生了什么,产出客观事件。它的依据是客观世界的进程,不是角色活不活跃——时间到了下课就会发生,跟 life 产没产生行动无关,所以不会自锁。它读 life 状态只为"让客观叙述跟她对得上"(她在上课就别说她在街上),绝不为"判断要不要叫她";这两个用途必须切开。它只陈述客观事实,不把谁的内心活动 / 情绪变成世界事件。

**2. life = 纯事件反应者。** 只被事件激活,激活后基于最新处境决定这一刻干什么、产出真实生活记录(落库)。跑完就等下一个事件,自己绝不设闹钟、不排下次。她的自主性靠 prompt 给(引导她更主动地决定做什么),不靠机制。她不主动上报状态给 world,只把状态写库,world 下一轮自己 pull(沿用现状 pull 范式)。

**3. 闹钟和日程是两回事,删闹钟、用日程。** 闹钟(next_wake_at)是个空的时间点,目的只是"维持她运转",一旦她的存活依赖它,设错 / 丢失就永久睡死——存活不该压在她自己手上,删。日程是她真实生活里有内容的安排(答应明天交作业、打算今晚复习),到点提醒她去做,丢了顶多"她忘了做某事"(真实、可接受的生活后果),不是睡死。她的主动计划全走日程,日程现状已有(notebook + 到点提醒),复用。

**4. 存活由世界的客观事件流兜底,不由 life 自己。** 时间照走、下课照发生、天照黑,这些客观事件持续把在场的 life 卷进生活,所以她不需要闹钟也不会永久静止。整套里唯一必须用机制死保的,只有 world 的保底心跳——一个后台定时循环,改成投出信号就立刻进下一拍、绝不等下游、绝不被卡住(现状 source loop 是同步等下游,会被一轮卡死的 LLM 堵停)。这个循环不用模型、不做判断、不花钱。

**5. 世界节奏 = 保底心跳兜底 + world 自排加密。** 保底心跳是固定的最低频率(绝不睡死);world 跑完一轮自己排一个"多久后再激活我"的延时点,平淡排远、热闹排近,用来省钱。world 自排和被删的 life 闹钟的根本差别:它只是在保底心跳之上加密,排丢了退回保底心跳那么稀、不会死;不是唯一唤醒源。现状已有雏形(interval 心跳 + emit_delayed 自排)。

**6. 事件投递 = 客观作用域 + 在场匹配。** world 产出客观事件时,顺手标这事的客观作用域(发生在哪、广播性的还是指向某个具体的人)——这是它客观描述的一部分,不是"我要推给谁"的决定。然后一道轻量模型判断做在场匹配:拿事件作用域对一下每个角色此刻客观在哪,谁在场谁收到("下课"全校广播 → 在学校的三姐妹都收到;"妈妈喊绫奈吃饭" → 在家的绫奈收到)。这道判断只看"事件发生在哪 + 谁在那",不看"谁该不该被叫醒",所以不自锁;它很轻(输入是一条事件作用域 + 几个角色的位置)。它替换现状 world 主观挑收件人(notify recipients)的动作。

## 跟现状的本质区别

不在零件上(保底心跳、world 自排、事件队列、PG、外部源、NPC、日程,现状都有),在三个职责的归属变了:

- world:角色唤醒管理者(主观决定叫谁、会自锁)→ 纯客观世界推演者(只陈述发生了什么,不管谁醒)。
- life:被 world 主观叫醒 + 自设闹钟 → 纯客观事件驱动 + 日程承载主动计划 + 存活靠世界兜底。
- 收件人:world 主观挑(notify recipients)→ 客观作用域 + 在场匹配。

## 基于当前分支的改动(改 / 删 / 新 / 留)

- **改**:source loop 改成 fire-and-forget(投出心跳不等下游,绝不被卡死);world 去掉"读角色状态判断该叫谁"那段,改成只产客观事件 + 标作用域;事件投递从 world 主观选收件人改成在场匹配。
- **删**:life 的 next_wake_at / fire_life_self_wake 自设闹钟整条。
- **新**:客观事件的作用域标注;在场匹配那道轻量判断;若现状没有,补角色"此刻在哪"的客观状态。
- **留**:保底心跳、world 自排(emit_delayed)、EventArrived → life_wake 入口、日程(notebook + 到点提醒)、act / pull、外部源 daily_fetch、NPC(npc_visit)、chat、day_review / persona_review 两个 cron。

## 待确认 / 实现时定的点

- 角色"此刻在哪"现状 life 状态里有没有、够不够明确(在场匹配的依据;没有就要补,且由 life 自己写、不硬编码)。
- 模糊在场(下雨,在室内还是室外、看没看见窗外)那道轻判断怎么处理。
- 保底心跳间隔、world 自排的典型间隔取值。
- chat 小改:把"这是多久以前的她"如实告诉模型,别用现在时态假装她一直在忙(现状穿帮源)。

## 数据 & 部署影响

- 可能新增 / 调整 life 状态里的"位置"字段(durable Data 改动要走 framework 注册 + schema migrate,coe 有旧表需注意 fail-closed)。
- 不新增 prompt 变量契约破坏;world / life 的 prompt 要按新职责重写(world 收口成纯客观、life 加自主性引导),走 Langfuse 泳道发布验证。
- agent-service 单服务部署;部署会中断正在跑的后台世界推演,部署前确认无在跑任务。
- 先 coe 验证再上 prod;coe 注意冷启清库 + MQ 遗留旧 schema 消息那几个老坑。

## Tasks(粗颗粒)

1. **world 收口成纯客观推演者**。目标:world 不再读角色状态做唤醒决策、不再主观挑收件人,只基于真实时间 + 外部信息产出带客观作用域的客观事件。产出:改写后的 world 推演逻辑 + prompt。验证:coe 跑一段,确认 life 全程静止时 world 仍按客观节奏持续产出事件(不再自锁),且事件内容是纯客观陈述。

2. **删 life 自设闹钟,改成纯事件反应 + 强化自主性**。目标:删 next_wake_at / self-wake 整条,life 只被事件激活、跑完不排下次,prompt 给足自主性。产出:改写后的 life 唤醒入口 + prompt。验证:coe 确认 life 不再有任何自设唤醒,且被客观事件激活后反应自然、主动性足够。

3. **事件投递改成客观作用域 + 在场匹配**。目标:world 产事件标作用域,新增一道轻量模型判断按角色客观位置匹配收件人,替换 notify recipients。产出:作用域标注 + 在场匹配环节。验证:构造下课 / 点名 / 下雨等事件,确认投递到的角色符合客观在场,且这道判断不依赖角色活跃度。

4. **保底心跳改 fire-and-forget(source loop 解耦)**。目标:时间推进循环投出信号即返回、不等下游,任何一轮下游卡死都不停摆。产出:解耦后的 source loop。验证:故意让一轮 world 推演挂死,确认保底心跳照常推进、世界不睡死。

5. **存活与节奏端到端验证**。目标:确认整套在 coe 真机上撑住连续性、成本、不降质。验证:coe 连续跑数天,看 life 不再出现长时间系统性静止、日成本 < 100 美元、输出质量不随运行时长退化;chat 读到的是连续真实的生活记录。
