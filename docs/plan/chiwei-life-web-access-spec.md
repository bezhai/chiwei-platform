# 给 life 一只能上网的手:她想查就查、想刷就刷,做的事有真来源

## Problem

赤尾会说她"在刷手机""在看小说",但看完能聊出来的东西一眼假——观后感、刷到的内容,全是模型当场编的。根因不是她话少,是她**根本没真看,产出没有真来源,只能瞎编**。这跟之前"高考硬编剧情进 prompt"被否是同一个病:假的来源,只能产出假的东西。

而现在 life 里的角色,所有行动都是语言形式、由 world 推演,她手里**没有任何真去网上拿东西的能力**——world 的眼睛能上网(抓天气、番剧),但那是给整个世界抓共享底料的,不是她个人按自己当下的需要去拿。

## Goal

给 life 加上"能上网拿真东西的手",她在自己的生活循环里自然地用:

- **带着问题查**:她心里有件具体的事(明天要去约会),为这件事需要知道某个信息(明天下不下雨、餐厅几点关),自己带着问题去查、当场拿到真答案,基于真数据反应(真知道该不该带伞,而不是编一句"明天应该不错")。
- **没事刷手机**:她没具体问题,就想刷刷看有啥,跟她当下兴趣相关的真内容涌到她面前,翻到感兴趣的才停。刷回来的是真资讯、真话题、别人真在聊的,够她产生真实反应——可能顺手记进本子,可能下次聊起来带出来。

靠工具说明引导她自然地用(跟刚上线的本子一样,没人教她也会用),不写死刷的时间、不写死她喜欢啥。

## Non-goals

- **不做"撞见世界大事"**(社会新闻、热搜冲击):那是 world 自己够不够丰富的问题——world 推演的世界够厚,大事自然顺着她的感知流到她那儿;不够,在外面单加个"大事推送"只是打补丁、还违背设计原则。归 world 那条线,不在这次。
- **刷手机不是世界要闻入口**:她刷到的是自己兴趣圈里的资讯和讨论(新番、某个领域的新鲜事),不是时政、社会突发、灾害预警那种世界级大事——后者该不该进她意识,归 world 推演,不从 life 这边偷渡(否则等于绕过上面那条、还可能踩信息差红线)。
- **不做信箱 / 敲门被动推送**:这次的"查"和"刷"都是她主动发起的,不需要"内容被投进信箱再敲门唤醒她"那套机制。
- **不做"真把小说 / 视频看完"**:那是后面"真做事 = 一件事一个 agent 工具"的长线;这次只到"查到、刷到资讯和话题"这一层,够她产生真实反应即可。
- **不写死兴趣标签**:不硬编 anime / career / gaokao 这种枚举,她喜欢啥从她自己的信号长出来。
- **不单开"订阅更新"**:她惦记的东西更没更,刷手机时自然会带到,不单做一条。

## Key design decisions

1. **给 life 工具,不搞后台 agent 投信箱。** 砍掉被动推送(world 大事)之后,剩下的"查"和"刷"都是她主动发起的——她在自己的 life 工具循环里调、当场拿结果就行,不需要信箱 / 敲门。选同步工具而非异步抓取投递,因为人查天气、刷手机都是当场拿到、当场反应,不是"内容涌进脑子里再处理"。

2. **"查"和"刷"是两种用法,底层都是她 life 主 agent 调 search_web。** 查:她心里有具体问题,带着问题直接找、求一个答案。刷:她没有标准答案要求、就是去逛——由她自己(life 主 agent)读自己当下的上下文、自然涌出"想看看 XX"的兴趣方向,调工具捞一批真内容回来浏览,翻到感兴趣的才停。关键区别在她心里带的是"具体问题"还是"兴趣方向",不在底层能力。**刷这只手不设"必填关键词"的硬参数、不在工具里写死规则去猜她喜欢啥、不另起一个 agent 替她判断**:一旦逼她憋检索词,就丢了"内容涌来"的味儿;一旦在工具里内置兴趣规则或叠个子 agent,就违了宪法。她的兴趣方向是读自己状态后的自然流露,不是工具强加的检索契约。

3. **兴趣 emergent、不写死,信号取自她 life 轮本就有的上下文。** 她"想逛点啥"从哪来——取她这一轮里本就稳定可用的信号(以实现时确认为准:本子里还惦记的事、上一页日子、当天意识流 / 心情),不硬编 anime / career / gaokao 这种标签枚举。她 life 本身就是个懂她的 agent、这些信号又都在她当轮上下文里,所以让她自己想该刷啥即可,不在工具外面再叠一层 agent。**实现前先确认这些信号在 life 轮上下文里确实可用**,别假设有"最近几天日记 + 实时心情"这种其实不稳定的东西,免得实现者各自猜。

4. **她拿到的真东西要带着来源进她当轮上下文,但不单建表。** 查到 / 刷到的结果不能是"模型消化后的一段话",得带着源证据(标题、来源、时间、关键摘录)进她这一轮的上下文,这样她的反应才真挂在真东西上、聊起来不假。她想长期留住的,自己记进本子(本子会进她后续唤醒输入、由本子机制保全);没记的就像人一样慢慢忘,这是对的、不是 bug。不单建表,是因为"她刷到啥"是过程、不是需要长期检索的状态;langfuse trace 只作我们排查"她当轮真查了啥"的证据,不是她的回忆材料。

## Caller coverage

- **`build_life_tools`(app/nodes/life_tools.py)**:life 的工具集,这次加新工具。现状已有 act / note / edit_note / read_notebook / schedule 等;加"上网的手"是新增,不动现有工具的行为。这两只手**只进 life 的工具循环**;真人 chat 即时回复走 `agent_stream` 独立的 `ALL_TOOLS`,不会冒出这两只手。聊天结束后回灌信箱、唤醒的后续 life 轮会看到它们——这是预期副作用(她聊完可能去查 / 刷),不是回归。
- **`search_web`(app/agent/tools/search.py)**:现在给 world 的眼睛和 chat 用。life 复用同一个底层能力,**不改它的签名、不影响现有调用方**;复用 external_sources 里的结构化查询(天气等)同理。
- **world 的眼睛抓取链路 + `DailyMaterials`**:不动。那是世界共享底料,跟她个人查 / 刷是两回事,两条互不干扰。
- **life 唤醒拼 stimulus(app/nodes/life_wake.py)**:她查 / 刷在工具循环里就地反应,**不改唤醒输入的拼装**(不像本子那样要往唤醒里注入"还没了结的事")。她去用这两只手,靠工具说明引导,不改 wake 的 prompt 结构。

## Data & deployment impact

- **表**:倾向不新建(见决策 4)。若实现中确实发现需要留痕一份"她刷到的",再轻量评估,spec 不预设。
- **prompt**:不新增 langfuse prompt。引导她用这两只手靠工具的 docstring(代码里),不是 langfuse 模板。"刷"内部若要 LLM 帮忙组织呈现,优先直接用 search 结果、不另起一个 LLM 环节;真要起再单独评估。
- **部署**:agent-service。常规重启会重启 life 自驱动循环,无额外后台任务被杀。
- **外部依赖**:search_web 底层(You Search / Google CSE)、external_sources 结构化工具,都已在用,无新依赖。

## Tasks

1. **给 life 一只"带问题查"的手**
   - 目标:她生活里冒出具体目的时,能自己带着问题去网上查、当场拿到真答案。
   - 产出:life 工具集里一个让她传入自己想好的问题、主动检索的工具(底层走现有 search_web,可含天气等结构化查询);返回带着来源进她当轮上下文。
   - 验收:coe 上制造一个让她产生目的的情境(如聊到明天约会),在 langfuse trace 里看到她 life 轮真调了这个工具、问题是她自己想出来的、返回是真数据且带着来源进了她当轮上下文,且她后续反应建立在真数据上而非编造。

2. **给 life 一只"刷手机"的手**
   - 目标:她没具体目的时能刷手机,跟她当下兴趣相关的真内容涌到她面前,她翻到感兴趣的才停、基于真内容反应。
   - 产出:life 工具集里一个"刷手机"工具——不要求她预设关键词、工具内不写死兴趣规则、不另起 agent;她带着自己读状态后涌出的兴趣方向去用,工具捞一批带源的当下真内容回来供她浏览。
   - 验收:在她**没有外部目的的 self-wake 空闲轮**里(不是飞书诱导出来的),trace 看到她自然调了刷手机、捞回的是真资讯 / 话题、贴她当下兴趣、内容带着来源进了她上下文、她基于真内容反应;且刷到的是她兴趣圈的东西、不是时政社会要闻(那归 world)。

3. **闭环 + 真机验证**
   - 目标:确认两只手在她真实生活里被自然地用、产出有真来源不"一眼假",且没破坏现有 life 唤醒 / 工具循环。
   - 产出:coe 泳道部署 + 现有 life 测试通过 + 红-绿测试覆盖两只手的工具行为。
   - 验收:coe 跑一段,给出 trace 证据——飞书聊天带出目的那轮她真去查、**没有外部目的的自驱动空闲轮里她也会自然刷手机**(证明 docstring 引导对消遣行为也成立)、查 / 刷拿到的都是带源真东西、反应基于真内容、现有 self-wake / 本子等行为不受影响。

> 三个 task 有先后:先做两只手,再闭环验证。
