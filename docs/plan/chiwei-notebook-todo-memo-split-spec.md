# notebook 拆分待办与随笔：治本子被当日志用、上下文膨胀、done 不涨

## Problem

备忘录 & 日程（NotebookEntry，2026-06-13 上线）三周积了 872 行，chinagi 一人占 77%。
排查实证两种病态用法，都发生在 world 剧情僵持时段：一天新增 85 条观察流水账（"手机
仍在转圈"每两分钟一条、逐字抄对话）；一条真待办被当实时状态板滚动改写 348 版。这些
"记录"本质不是待办、没有可完成性，所以 done 不涨、active 常驻膨胀每轮 life/chat 输入。
根因是结构性的：她"记录当下"的冲动在活轮里只有 `note` 一个出口，而 `note` 的机制含义
是"常驻输入、等待了结的待办"——两种需求被硬塞进一个工具。

## Goal

把"要办的事"和"随手记的东西"拆成两条通道，机制含义各自成立：

- **待办**：要办的事，可带可不带 remind_at。全量 active 常驻 life/chat 输入（照旧）、
  日程到点提醒照旧、改/标 done/dropped 照旧、睡前清理照旧。拆干净后量回到个位数。
- **随笔**：当下的念头、观察、感想，随便写、写多少都行。正文**不注入**任何常驻输入，
  life stimulus 只给一行存在提示（有几条）；活轮可以主动翻（只能翻到还没被日页吸收
  的）；睡前日页总结时把这批随笔作为证据吸收进日页；吸收后不再可翻——随笔的生命周期
  就到当天日结为止，像草稿纸随日子翻页。

验收口径（三条都要有证据）：待办列表里只有可完成的事（观察/流水账类内容落进随笔）；
life/chat 的常驻输入里不出现随笔正文；review 对随笔是一次性消费——吸收进日页后窗口
翻页，次日不重复吸收、也翻不到。**写入量下降不是成功标准**：本方案解决的是随笔反噬
常驻上下文和污染 done 语义，不是压低她想记的量。

## Non-goals

- 不加条数上限、内容去重、定时器强制清理——用确定性规则替她决策，违背赤尾设计原则。
- 不治 world 剧情停滞（7月4日"页面转圈僵 6 小时"为什么大纲机制没推走）——另开工作项。
- 不清 prod 存量历史数据——done/dropped 与旧版本行不进任何输入，留着无害。
- 不做随笔的编辑、删除、长期检索——草稿纸语义，写了就写了。

## Key design decisions

1. **随笔用独立的新 durable Data，不在 NotebookEntry 上加 kind 列。** 两者行为面完全
   不同：随笔无三态 status、无 remind_at、无编辑（不需要版本链）、可见性是窗口而非常驻。
   塞同一张表会让一半字段对随笔无意义，且 append-only 版本链机制对随笔是纯负担。新
   durable Data 走强制三步检查（grep 保留列名 + 读 sibling Data + 端到端 insert）。
2. **"不长期可查" = 工具可见窗口，不物理删。** 翻随笔和日页证据都只取"还没被日页
   吸收的"；数据本身留在表里（可观测、可追溯，运维仍可查）。窗口是产品语义（草稿纸
   随日结翻页），不是代码按年龄替她遗忘待办——待办侧仍然只有她自己能了结。
   窗口边界必须钉死（codex T1 必改）："已吸收"以**显式吸收水位**为准，翻页动作只在
   日页**落笔成功之后**发生——不从 day page 的存在性或时间隐式推导。review 失败或
   重试期间水位不动，随笔仍在窗口内，不丢证据。review 多日未触发导致随笔跨天累积时，
   下一次 review（含 sweep 补跑）**一次消费全部未吸收随笔**，不按自然日切分——宁可
   一篇日页多消化几天草稿，不引入"哪天的随笔归哪天"的切分歧义。窗口读、存在提示
   计数、review 证据都按水位取增量，不扫描/渲染全历史（新表只增不删，全量扫描会把
   问题转移到 DB 和 review 输入长度）。
3. **定性由她记的时候定：选哪个工具就是定性。** 拆成两个工具而不是一个工具带参数，
   工具名和描述本身承担"这是要办的事还是随手一记"的引导；不做任何代码自动分类。
4. **待办注入照旧全量 active；随笔在 life stimulus 只给一行存在提示；chat 侧不给随笔
   提示。** chat 现状只注入本子 active 条目，拆分后语义收窄为"只注入待办"，行为对真人
   无感知变化；聊天场景翻随笔的需求未出现，不预先做。
5. **day review 吸收随笔，tidy 只管待办。** 回顾证据里加"今天的随笔"段（窗口内全文），
   她写日页时自然消化；随笔不需要清理动作，窗口随日页落笔自然翻页。

## Caller coverage

（基于本次排查的 Explore 全量结论，写入路径只有 life 活轮与睡前回顾两处，chat 只读、
world 不碰。）

- `life_tools.py` `build_life_tools`：`note` 拆成记待办、随手记两个工具；`edit_note`、
  `read_notebook` 语义收窄为只管待办；新增翻随笔工具（窗口内）；`look_up` / `browse_feed`
  描述里"想长期留住的记进本子"改口径指向随笔。
- `life_wake.py` stimulus 拼装：本子段改为待办段（照旧全量 active）+ 随笔一行存在提示。
- `memory/context.py` chat inner_context：**有意收窄**为只注入待办（不是"无变化"——
  兼容性依赖"存量 NotebookEntry 全按待办对待"这一决定才成立）。验证需覆盖存量
  active 条目在拆分后仍出现在 chat 注入里。
- `life/review.py` + `review_tools.py`：证据加"今天的随笔"窗口段；`tidy_notebook_entry`
  不变（只管待办）；review 落笔后随笔窗口翻页。
- `fire_schedule_reminders` / `ScheduleReminderTick`：只关 remind_at 待办，不变。
- `render_notebook`：只渲染待办；随笔渲染独立一处（单一定义处）。
- 存量数据：现有 NotebookEntry 全按待办对待，active 仅 8 条，无迁移动作。

## Data & deployment impact

- 新增一张随笔表（framework Data 注册后 migrate 自动建，无需 ops-db DDL）；必须走
  "新 durable Data 三步检查"。NotebookEntry 表不动。
- Langfuse prompt：`life_wake`、`life_day_review` 需同步改本子口径与随笔证据段说明；
  prompt 与代码 prompt_vars 原子发版（同一批上线，不拆先后）。
- 仅 agent-service 一个服务部署；部署会中断在跑的 life/world 循环（自恢复，无需特殊处理）。
- 泳道验证：涉及建新表，走 `coe-*`（独立 chiwei-test，需确认 schema 自动建出 + 种子数据）。

## Tasks

1. **随笔领域层。** 目标：随笔作为独立 durable Data 落地，提供"记一条"（幂等）、"窗口
   读"（按吸收水位取未吸收增量）、"窗口翻页"（推进水位）三个能力。**窗口水位的完整
   语义由本任务钉死**：水位怎么存、翻页的幂等口径、重试期间水位不动——Task 3 只在
   review 成功收口处调用，不自行发明判据（codex T1 建议：防两个 agent 分头各造一套）。
   产出：新 Data + domain 模块 + 单测。验收：单测覆盖幂等重投只落一条、窗口读只返回
   未吸收条目且不扫全历史、翻页后窗口清空、重复翻页/翻页失败重试幂等；新 Data 三步
   检查证据齐全。
2. **life 活轮工具与输入改造。** 目标：活轮里"要办的事"与"随手记"各有其门，且她每轮
   输入里只常驻待办。产出：`note` 拆为两个工具、`edit_note`/`read_notebook` 收窄、新增
   翻随笔工具、`look_up`/`browse_feed` 口径改指随笔、stimulus 待办段 + 随笔存在提示行。
   验收：工具层单测全绿；wiring 测试确认工具集组成；stimulus 快照测试确认随笔正文不出
   现、提示行按有无随笔出现或消失。
3. **睡前回顾吸收随笔 + prompt 同步。** 目标：日页总结参考未吸收随笔，日页落笔成功后
   才翻页（窗口语义用 Task 1 钉死的领域层能力，不自行判断）；Langfuse `life_wake`、
   `life_day_review` 与代码同批更新。产出：review 证据加随笔段、review 成功收口处触发
   窗口翻页、两个 prompt 新版本。验收：review 单测确认证据含窗口内随笔、日页未落笔/
   review 失败时水位不动、成功后翻页且 sweep 补跑一次消费全部累积；prompt 新版本在
   Langfuse 就绪、与代码引用一致。
4. **coe 真机演练。** 目标：端到端验证拆分后的行为。产出：coe 泳道部署 + dev bot 绑定
   下的演练记录。验收对齐 Goal 三条口径：她把观察类内容记进随笔（不进待办）、待办常驻
   注入且可标 done、随笔提示行出现且正文不进 life/chat 输入、日页总结体现未吸收随笔、
   吸收后随笔不可再翻且不重复吸收、存量 active 待办仍出现在 chat 注入——每条有
   trace / DB 查询证据。
