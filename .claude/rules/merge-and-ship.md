# 合码与 Ship 铁律

## 合码必须等用户确认

合并 PR 前必须等用户明确说"合"或"merge"，不得自行判断"改动小可以直接合"。
无论改动大小，包括"只改了文档/配置"的情况，执行 `ghc pr merge` 前都要先告知用户 PR 内容并等待明确许可。

## Ship 前必须列出所有改动

1. **不允许遗漏提交**：ship 前必须列出分支上的所有 commit 和改动文件清单。如果有意料之外的文件改动（如 Makefile、基础设施文件），必须向用户确认是否一起合入，不能擅自跳过。
2. **合并冲突必须用户确认**：rebase/merge 产生冲突时，必须先展示冲突内容给用户，让用户决定怎么解决，不能自行选择版本。基础设施文件的冲突解决如果选错版本会影响部署。
